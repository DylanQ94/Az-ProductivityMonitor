# Azure Databricks Processing Layer

## Overview

Azure Databricks was used as the main processing and transformation engine for the Productivity Monitor Lakehouse on Azure project.

This phase implements the Medallion architecture over Azure Data Lake Storage Gen2, transforming raw productivity monitoring files into clean, analytical Delta Lake tables.

Processing flow:

```text
ADLS Gen2 Landing
        ↓
Bronze Delta Layer
        ↓
Silver Delta Layer
        ↓
Gold Analytical Layer
```

## Purpose of the Databricks Layer

The goal of this phase was to transform daily CSV and JSON files generated by the productivity monitoring application into structured Delta Lake datasets.

Databricks was responsible for:

- Reading raw files from ADLS Gen2.
- Creating a Bronze Delta table with source metadata.
- Cleaning and normalizing the data into a Silver Delta table.
- Converting cumulative counters into real interval-level duration metrics.
- Building a Gold analytical model with fact and dimension tables.
- Preparing the data for a SQL serving layer and future reporting consumption.

## Bronze Layer

The Bronze layer stores the source data with minimal transformation.

Input path documented in the Databricks phase:

```text
landing/source-desktop_app/
```

Output path:

```text
bronze/productivity_events/
```

The Bronze notebook reads CSV and JSON files recursively from ingestion-date folders such as:

```text
landing/source-desktop_app/ingest_date=YYYY-MM-DD/
```

Original source columns:

```text
timestamp
level
face_front
input_active
session_seconds
attentive_seconds
productive_seconds
activity
```

Additional technical metadata columns were added for lineage:

```text
_source_format
_source_file
_source_file_name
_ingestion_timestamp
_ingest_date
```

During development, the Bronze table was written using overwrite mode to avoid duplicated records when re-running the notebook.

## Silver Layer

The Silver layer cleans, types, and standardizes the Bronze data.

Output path:

```text
silver/productivity_events/
```

The Silver transformation includes:

- Timestamp parsing.
- Date and time column creation.
- Numeric type casting.
- Boolean conversion for face and input activity signals.
- Activity text cleaning and standardization.
- Productivity state creation.
- Interaction state creation.
- Invalid record filtering.
- Deduplication using a technical `record_hash`.

The `record_hash` column was created using SHA-2 hashing over key record attributes. It is used to identify duplicated records and support reliable downstream processing.

The Silver layer defines the following productivity state logic:

```text
level = 0 → inactive
level = 1 → partial_active
level = 2 → fully_active
```

## Gold Layer

The Gold layer was designed as the analytical model of the project.

Instead of creating only direct summary tables, the final design uses an interval-level fact table as the source of truth:

```text
gold/fact_productivity_interval/
```

This table is built from Silver and calculates real row-level duration increments using window functions and `lag()`.

The source duration fields are cumulative counters:

```text
session_seconds
attentive_seconds
productive_seconds
```

Because of that, direct summation would overestimate time. The Gold interval fact calculates real increments by comparing each row against the previous row from the same source file.

The final duration model is:

```text
session_duration = attentive_duration + productive_duration + inactive_duration
```

Inactive duration is calculated as:

```text
inactive_duration = session_duration - attentive_duration - productive_duration
```

The Gold interval fact includes:

- Analytical keys.
- Natural date and category columns.
- Original cumulative counters.
- Real duration increments in seconds and hours.
- Interval-level ratios.
- Record-level flags.
- Data quality indicators.
- Source lineage columns.

## Gold Fact Tables

Final Gold fact tables:

```text
fact_productivity_interval
fact_productivity_monthly
fact_productivity_daily
fact_productivity_hourly
fact_productivity_activity
fact_source_file_audit
```

The aggregated fact tables are built from `fact_productivity_interval`, not directly from Silver. This ensures that all aggregations reuse the same validated interval logic.

Main metrics:

```text
total_session_hours
total_attentive_hours
total_productive_hours
total_inactive_hours
total_active_hours
attentive_ratio
productive_ratio
inactive_ratio
active_ratio
```

## Gold Dimension Tables

Final Gold dimension tables:

```text
dim_date
dim_month
dim_hour
dim_activity_category
dim_source_file
```

These dimensions provide reusable analytical context for reporting and future Power BI modeling.

The model uses keys such as:

```text
date_key
month_key
event_hour
activity_category_key
source_file_key
```

These keys allow the Gold facts to be related to the corresponding dimensions in a star-schema-like analytical model.

## Final Gold Structure

```text
gold/
├── fact_productivity_interval/
├── fact_productivity_monthly/
├── fact_productivity_daily/
├── fact_productivity_hourly/
├── fact_productivity_activity/
├── fact_source_file_audit/
├── dim_date/
├── dim_month/
├── dim_hour/
├── dim_activity_category/
└── dim_source_file/
```

## Design Decisions

A productivity-state dimension and an activity-state record fact were considered during the design process, but they were removed from the final model because the project prioritizes duration-based productivity analysis over record-count-based state analysis.

Aggregated facts are built from `fact_productivity_interval` to ensure that all duration aggregations are based on real interval increments rather than cumulative counters.

## Output of the Databricks Phase

At the end of this phase, Databricks produced a curated Gold Delta Lake model ready to be exposed through a SQL serving layer.

The next phase creates Synapse Serverless SQL views over the Gold Delta folders so they can be consumed by analytical tools.
